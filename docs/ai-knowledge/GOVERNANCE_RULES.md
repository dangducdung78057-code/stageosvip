# StageOS 治理规则

## Git 是事实来源

- Git 仓库是代码、配置和文档状态的唯一事实来源。
- 任何没有进入 commit 的本地文件都不属于可恢复基线。
- 状态报告必须给出 Repository、Branch、HEAD 和 Working Tree 状态。
- 不以聊天记录、构建目录或旧导出包替代 Git 历史。

## main 分支规则

- `main` 表示可追踪的主开发基线。
- 修改前先核对分支、HEAD 和工作树，避免覆盖无关变更。
- 禁止 force push、历史重写和未经确认的直接合并。
- 提交应保持单一目的，并在提交前检查实际 diff。

## Tag 使用规则

- Tag 用于标记经过人工确认的重要恢复点或发布基线。
- `stageos-v0-import-baseline` 是 v0 导入基线，不得移动、覆盖或删除。
- 新 Tag 必须指向明确 commit，并在推送后核对本地与远端引用。
- Tag 不是功能已上线或 Supabase 已部署的证明。

## AI 修改权限

- AI 只能在用户明确授权的目录、功能和操作范围内工作。
- 修改前必须检查现状，修改后按风险执行类型检查、测试或构建；用户禁止时不得执行。
- AI 必须保留用户已有改动，不得擅自清理工作树。
- 涉及 commit、push、部署、外部消息或破坏性操作时，必须有明确授权。
- 未实现、部分实现和已实现必须如实标记为 `planned`、`partial`、`implemented`。

## 禁止事项

- 禁止伪造测试、构建、部署、支付或远端鉴权成功状态。
- 禁止提交密钥、Token、真实学生姓名或真实面部资料。
- 禁止使用 `reset --hard`、强制推送或未经确认的批量删除。
- 禁止从旧 `.git`、`node_modules`、`dist`、缓存或临时文件恢复开发历史。
- 禁止整体替换生产主线，或把参考实现直接冒充正式实现。
